DHCP Snooping 配置

Posted on 2018-01-02 | Edited on 2019-02-21

什么是DHCP？

做为网络的基本规则之一，网络上每一个节点都需要地址才能互相通信。在早期时候，因为网络设备很少会移动，通常是静态分配地址的。但是静态地址分配没有扩展性。想象一下每次你离开家时都要给你的智能手机改一下地址。因此，现在动态地址分配已经是全世界主要网络的事实标准了。几乎所有地方都在使用DHCP（Dynamic Host Configuration Protocol，动态主机配置协议）给电脑，手机，平板等各种设备分配地址。

在各个网络中都会有一台合法的DHCP服务器给网络中的设备分配地址。如果在网络中接入一台非法的DHCP服务器，此非法DHCP服务器会跟合法DHCP服务器争抢，造成终端设备无法取得正确配置而无法访问网络。

大部分情况这种非法的DHCP服务器都是因为意外连接上网络的。比如某人想在其工位上扩展无线连接，拿一个默认开启DHCP服务的无线路由器直接连上网络。他自己也没有意识到这个无线路由器带来的问题。

同时，攻击者也利用这种DHCP的开放性，把自己插入在某台设备与其要访问的资源之间，进行中间人攻击窃取用户的隐私和密码。

DHCP Snooping是一种用来防止这种安全风险的两层交换技术。

为了更好的理解这个DHCP Snooping，我们先来看看DHCP是怎样工作的。

DHCP如何工作？

首先当一台设备上线时，以自身MAC地址为源向全网广播一个DHCPDISCOVER消息，询问是否有DHCP服务器。

当DHCP服务器收到DHCPDISCOVER消息时，向发送设备返回一个包含地址租约的DHCPOFFER消息。

设备收到DHCPOFFER后，向DHCP服务器响应一个DHCPREQUEST消息。如果有多个DHCP服务器发送DHCPOFFER消息，终端设备只接受收到的第一个DHCPOFFER。

最后DHCP服务器发送DHCPACK消息给终端设备确认地址租约。此时终端设备开始可以进行通讯了。

mGRE - multipoint GRE tunnel

Posted on 2014-06-13 | Edited on 2019-02-21

Features

  1. tunnels between each two sites, traffics between spoke sites don’t need to pass the hub site.
    • permanent tunnel between Hub site and each Spoke site
    • dynamic tunnel between Spoke sites on demand
  2. simplify the configuration
  3. In Spoke site, static IP address is not required. static IP address is required only in Hub site.

How is it working

  1. Hub site works as a NHRP server and ready to accept the register from the spoke sites
  2. Spoke sites register their IP to the Hub site(NHRP server).
  3. permanent tunnels created between Hub and each Spoke.
  4. the data traffics between Hub and each spoke go thru the permanent tunnels.
  5. if there is any data required traffics from spokeX to spokeY, spokeX query from hub for the spokeY’s IP address.
  6. a dynamic tunnel created between spokeX and spokeY.
  7. the data traffics between spokeX and spokeY go thru the dynamic tunnel but no need to pass thru the Hub.
  8. while there is no data traffic between spokeX and spokeY, a timer counted down, the tunnel destroyed when expired.

来电显示

来电显示开通后，由于局端的设备不同，来电显示的制式有可能为DTMF或者FSK。FSK的来电显示，通常是在第1、2声振铃之间，局端同时会伴随有对时信息，如果终端设备有此功能，可以自动调整时间，但FSK也会有3种不同的制式应用在不同的国家。DTMF的来电显示多数会在振铃前。

多数电话机设备不太会有制式不兼容的问题，一般都可以自动检测到不同的制式并显示来电号码。（强烈要求思科自适应来电！！！）

所以，在确认电信的来电显示已开通（比如用话机可以显示号码）但路由器通过debug vpm all看到caller id receive failed时，就很有可能是制式的不兼容造成的，这时可以试试看改变cptone来进行调整并测试。不能太指望电信的接口人员会告诉我们详细的技术规范，至少我从来没有碰到过。

The voice path is established in the backward direction at the start of the RTP stream. The forward audio path is not cut through until the Cisco IOS gateway receives a Connect message from the remote end.

In some cases, it is necessary to establish a two-way audio path as soon as the RTP channel is opened, which is before the Connect message is received. In order to achieve this, issue the voice rtp send-recv global configuration command.

upgrading catalyst to new feature set

Posted on 2010-12-13 | Edited on 2019-02-21 | In Uncategorized

Error: The image in the archive which would be used to upgrade
Error: system number 1 does not support the same feature set.

I just got this error message while upgrading a Cisco Catalyst 3560 from 12.2(35)SE5 to 12.2(44)SE. The older version was IP BASE and the new version is ADVANCED IP SERVICES.

It seems that Cisco has added a check into there upgrade process to prevent you from accidentally changing the feature set during an IOS upgrade. This is a really nice sanity check since changing feature sets on a production switch could really ruin your day.

To bypass this check, add the /allow-feature-upgrade parameter to the archive download-sw command. According to the documentation, this feature is new as of 12.2(35).
